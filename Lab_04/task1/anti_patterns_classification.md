# Архитектурная классификация антипаттернов

## 1. Глобальное состояние приложения

**Тип проблемы**: Нарушение инкапсуляции.
Переменные объявлены через `var` в глобальной области, из-за чего любое изменение может повлиять на весь проект.

**Почему это плохо**:
Код становится непредсказуемым и трудно отлаживаемым.

**Решение**:
Создать единый объект состояния приложения или использовать модульную структуру.

**Предлогаемый подход**:
Module Pattern / App State Object

Пример:
```javascript
const AppState = {
  user: {},
  config: {},
  cache: {}
};
```

---

## 2. Дублирование логики проверки email

**Тип проблемы**: Нарушение принципа DRY.

**Почему это плохо**:
Изменение формата email потребует правки сразу в нескольких местах.

**Решение**:
Создать единую сервис проверки данных.

**Предлогаемый подход**:
Validation Service Module

---

## 3. Разброссанные настройки

**Тип проблемы**: Отстутствие централизованной конфигурации.

**Почему это плохо**:
Сложно переключать окружения (dev, prod).

**Решение**:
Вынести все настройки в отдельный конфигурационный модуль.

**Предлогаемый подход**:
Config Module

---

## 4. Глобальные обработчики событий

**Тип проблемы**: Нарушение принципа инскапсуляции UI.

**Почему это плохо**:
Любой внешний скрипт может вызывать эти функции.

**Решение**:
Инкапсулировать обработчики внутри UI компонентов.

---

## 5. Функция с избыточной ответственностью

**Тип проблемы**: Нарушение Single Responsibility Principle.

**Почему это плохо**:
Функция выполняет слишком много задач, что затрудняет ее тестирование и поддержку.

**Решение**:
Разделить на независимые сервисы: 
- Validation Service
- Storage Service
- UI Controller
