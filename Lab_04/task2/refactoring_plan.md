## План рефакторинга — Лабораторная работа 4.2

### 1) Цель
Перевести устаревший монолитный e-commerce код в модульную структуру без потери функционала и исходных значений (налог, доставка, валюта).

### 2) Основные проблемы legacy
- Глобальные переменные (cart, products, user и т.д.)
- Смешение ответственности (логика + storage + UI-логи в одних функциях)
- Дублирование расчётов/форматирования
- Прямая работа с глобальными данными

### 3) Стратегия
1. Создать единый глобальный объект `MyApp` (Namespace Pattern).
2. Скрыть состояние внутри модулей через IIFE (Module Pattern).
3. Выделить отдельные слои:
   - `Config` — настройки и константы
   - `Utils` — форматирование/валидации/время
   - `Storage` — чтение/запись localStorage
   - `Services` — бизнес-логика (Product/Cart/User/Order/App)

### 4) Ожидаемый результат
- В глобальной области видимости только `MyApp`
- Состояние корзины/товаров/пользователя недоступно напрямую
- Нет дублирующих функций расчета subtotal/total и форматирования
- Код проще поддерживать и расширять
